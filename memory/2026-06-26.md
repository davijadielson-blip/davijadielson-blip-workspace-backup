## 22:58 UTC — Correção definitiva da reversão de config (agentes/modelos)

**Problema:** A config de agentes (`agents.defaults.model.primary` e `agents.list`) revertia sempre para um conjunto antigo (DeepSeek/Gemini como primary, apenas 14 agentes, 6 C-Levels ausentes).

**Causa raiz:** O `openclaw exec-policy preset yolo` era invocado automaticamente (por cron ou hook) e, como efeito colateral, regravava a config a partir de defaults internos do Gateway, sobrescrevendo a lista canônica de 20 agentes.

**Solução aplicada:**
1. Cron `ensure-critical-integrations` permanece desativado (ele chamava `ensure-integrations.sh` que já escrevia a config correta, mas não impedia a sobrescrição)
2. Config canônica aplicada via `openclaw config patch --file ops/openclaw-agents-definitive.patch.json`
3. Exec-policy sincronizada manualmente com `openclaw exec-policy set --security=full --ask=off` (em vez de `preset yolo`)
4. Gateway recarregado
5. Runtime validado: **20 agentes, todos configure:true, todos openai-codex/gpt-5.5**

**Proteção:**
- `exec-approvals.json` não tem agent-specific overrides (está limpo)
- Patch canônico salvo em `ops/openclaw-agents-definitive.patch.json`
- Para alterar config no futuro: `openclaw config patch --file ops/openclaw-agents-definitive.patch.json` + possivelmente re-sincronizar exec-policy

**Lições aprendidas:**
- `exec-policy preset yolo` altera agents.list como side effect — usar `exec-policy set` com flags explícitas quando possível
- Sempre validar runtime com `agents_list` APÓS reload do Gateway, não apenas o arquivo JSON
- O `exec-approvals.json` não precisa de agent-specific entries quando todos seguem o default
